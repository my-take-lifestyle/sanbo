# 参謀 — 資産運用 意思決定支援アプリ（Phase 1〜4）

`資産運用参謀アプリ_設計書.md` v1.0 の Phase 1〜4 実装。
Vanilla JS + ES Modules の静的 PWA（ビルド不要）。**このフォルダ（`sanbo/`）自体が GitHub リポジトリのルートになる想定**です
（リポジトリ: `github.com/my-take-lifestyle/sanbo`、Public）。
schemaVersion は **4**（v1〜v3 のエクスポート JSON もインポート時に自動マイグレーション）。

## 構成

```
sanbo/                          ← このフォルダがリポジトリルート
├── index.html                  シェル（下部タブナビ5つ + ヘッダー⚙️で設定、Chart.js CDN）
├── manifest.json / sw.js       PWA（オフラインキャッシュ + Share Target）
├── .nojekyll                   GitHub Pages の Jekyll 処理を無効化
├── icon.svg
├── sample-data.json            テストデータ（設定 → サンプルデータを読み込む）
├── css/style.css                モバイルファースト（375px 基準）
├── js/
│   ├── app.js                  ルーティング・起動時処理（スナップショット / 為替 / トリガー評価 / Share Target 受け口）
│   ├── state.js                appState（付録A）・save/migrate（v1→v2→v3）・エクスポート退避
│   ├── ulid.js                 ULID 生成
│   ├── derive.js               評価額・配分・待機資金・トリガー判定・レビュー期限・放置検知・要アクション
│   ├── api.js                  外部 API アダプタ（Frankfurter / Finnhub、フォールバック付き）
│   ├── prompts.js               コンテキストパック（付録B）+ テーマレビュー（§3.4-3）
│   ├── parse.js                 貼り戻しパース（付録C + themeReview 拡張。失敗時は生テキスト保存）
│   ├── feed.js                 自動収集フィード（data/feed.json の fetch のみ。Phase 3）
│   ├── backup.js                File System Access API 自動バックアップ（Phase 3）
│   └── views/                  dashboard / portfolio / themes / intel / journal / models / settings
├── config/
│   ├── feed-sources.json       自動収集の取得元 RSS（自由に編集可）
│   └── feed-keywords.json      自動収集のフィルタキーワード（テーマ名・ticker。自由に編集可）
├── scripts/
│   └── build-feed.mjs          Actions が実行する取得・フィルタスクリプト（Node、fast-xml-parser 使用）
├── data/
│   ├── feed.json               Actions が生成。アプリが fetch するのはここだけ
│   └── feed-seen.json          重複配信防止用キャッシュ（アプリからは参照しない）
├── package.json / package-lock.json   scripts/ 用の依存（fast-xml-parser のみ）
└── .github/workflows/feed.yml  日次 cron + 手動実行のワークフロー
```

タブは設計書 §2.1 どおり5つ（司令部・資産・テーマ・インテル・ジャーナル）。設定はヘッダー右上の ⚙️ から開きます。

### `docs/feed.json` ではなく `data/feed.json` にした理由

設計書 §4.5 の例は「Pages が `/docs` を配信する」構成を前提にしていますが、今回は `sanbo/` フォルダ自体（`index.html` が直下にある）がリポジトリルートで、GitHub Pages は **main ブランチのルート（`/`）** から配信します。この場合 `index.html` と `data/feed.json` は同一オリジンの通常のファイルとして共存でき、`docs/` という特別な階層は不要なため `data/` としました。挙動は設計書の意図（Actions が公開情報だけを静的 JSON として配信し、アプリが起動時に fetch する）と同一です。

## Phase 2 の機能とテスト手順

### 1. 価格トリガー自動判定
価格更新（Finnhub 一括 / 保有・ウォッチの手動価格）のたびに `price_below` / `price_above` を判定し、
成立すると司令部の「要アクション」に ⏰ で表示。「確認済み」で消し込み、条件が一度外れて再成立すると再表示。

**テスト**: テーマ画面 → VRT を開く → 「現在価格（判定用）」に 75 を入力して保存
→ 司令部に「トリガー成立: VRT が 80 を下回りました」→ 確認済みで消える
→ 価格を 90 に戻して保存（条件解除）→ 再度 75 で保存 → 再表示される。

### 2. 意思決定ジャーナル
ジャーナルタブから起票。レビュー期限は起票日 +6 か月で自動設定（編集可）。
期限到来かつ結果未記入の判断が司令部に出ます。レビューでは「判断の質」と「結果」を分けて記録します。

**テスト（日付操作）**: 起票 → 一覧から開いて「レビュー期限」を過去日（例: 昨日）に変更して保存
→ 司令部に「判断レビュー期限」が出る → 結果を記入すると消える。
サンプルデータには期限到来済みの Decision（2026-01-08 NVDA）が入っています。

### 3. PWA Share Target（スマホ実機での確認手順）
1. アプリを **https で配信**（GitHub Pages 等。localhost 以外では https 必須）
2. Android Chrome でサイトを開き「ホーム画面に追加」で **PWA としてインストール**
3. 任意のブラウザ/ニュースアプリで記事を開き、共有メニューから「参謀」を選択
4. アプリが開き、URL が受信箱の「未トリアージ」に入っていることを確認

ブラウザだけで擬似確認する場合: `index.html?title=テスト&text=https://example.com/article` を開くと同じ経路が動きます。
（iOS Safari は Share Target 未対応のため、URL 貼り付け導線を使ってください）

### 4. テーマレビュー（四半期）
テーマ詳細 → 「📋 テーマレビュー」でプロンプト（thesis の反証材料・KPI 状況・エクスポージャー過不足）をコピー
→ AI の回答をインテル受信箱の貼り戻し欄へ → テーマの「レビューノート」への追記とインテル1枚に自動展開。

### 5. 放置検知
active テーマへのインテル紐付けが約3か月（92日）途絶えると司令部に警告が出ます。

### 6. 新NISA 枠管理
残論点6（設計書 §6.2）が未決のため **実装していません**。決定後に Phase 2.1 として追加してください。

## Phase 3 の機能とテスト手順

### 1. GitHub Actions フィード（自動収集）

**セットアップ**
1. GitHub で `my-take-lifestyle/sanbo` を Public リポジトリとして作成（未作成の場合）
2. このフォルダの中身をリポジトリルートとして push
   ```
   cd sanbo
   git init
   git add .
   git commit -m "Phase 1-3 実装"
   git branch -M main
   git remote add origin https://github.com/my-take-lifestyle/sanbo.git
   git push -u origin main
   ```
3. GitHub の Settings → Pages → Source を **Deploy from a branch / main / `/ (root)`** に設定
4. Settings → Actions → General → Workflow permissions を **Read and write permissions** にする
   （`feed.yml` が `data/feed.json` を自動コミットするために必要）
5. Actions タブ → 「Update feed.json」→ **Run workflow** で手動実行し、`data/feed.json` が更新されるか確認
   （日次 cron は 21:00 UTC = 06:00 JST。それまで待たずにテストできます）

**動作確認**: アプリ（`https://my-take-lifestyle.github.io/sanbo/`）を開き、インテル画面の「自動収集」に
新着が出ることを確認。ローカル動作確認では `node scripts/build-feed.mjs` を直接実行すれば
`data/feed.json` がその場で更新されます（このフォルダで実際に一度実行済み・動作確認済みです）。

**取得元・キーワードの変更**: `config/feed-sources.json`（RSS/Google ニュース検索フィードの URL）と
`config/feed-keywords.json`（フィルタキーワード）を編集して push すれば、次回実行から反映されます。
日経・ロイターは公式の汎用 RSS が廃止されているため、Google ニュースの `site:` 検索フィード（個人の
非商用フィードリーダー用途として許可されている形式）で代替しています。Bloomberg は公式 RSS を直接使用。

**鉄則の確認**: `config/`・`scripts/`・`.github/` のいずれにも、保有銘柄の数量・金額・比率・口座名は
一切含まれません（`grep` で確認可能）。含まれるのはテーマ名・ticker などの公開して困らないキーワードのみです。
`sample-data.json` は明示的なテストデータで、実データではありません。

### 2. 自動バックアップ（File System Access API）

設定画面 → 「自動バックアップ」。対応ブラウザ（Chrome/Edge のデスクトップ等）では保存先ファイルを一度指定すると、
以後は「今すぐ上書きバックアップ」ワンタップで同じファイルへ上書きできます。非対応ブラウザ（iOS Safari 等、
モバイルブラウザの多く）ではセクション自体が「非対応」表示になり、従来のダウンロード方式にフォールバックします。
ダッシュボードの30日警告は、手動エクスポートと自動バックアップの**新しい方**で判定されます。

**テスト（ラウンドトリップ）**: 自動バックアップ実行 → 設定 → 全データ消去 → 保存したファイルを
「JSON インポート」で読み込み → 復元を確認。

### 3. API キー直結の AI 要約・Actions 側の自動要約

いずれも**実装していません**（このアプリ運用ではコピペ運用に一本化する方針で決定済み）。

## Phase 4: マルチモデル選定エンジン（設計書 §5.2）

「モデル」= 構造化プロンプト + 仮想ポートフォリオ + 成績記録。自動売買・自動執行・学習は行いません。
**ピックは参考シグナルであり推奨ではありません。** ヘッダーの 📈 からスコアボードを開けます。

### 月次ループ（各10分）
1. 📈 → 「月次モデル実行」で各モデルの実行プロンプトをコピー → AI チャットに貼る
2. 回答（`modelPicks` の JSON フェンス）を貼り戻し欄へ → ModelPick として記録
   （エントリー価格は保有/ウォッチの既知価格 → Finnhub キーがあれば自動取得の順で補完。日本株は手動入力）
3. 「米国株価を一括更新」でピック現在価格と SPY ベンチマークも自動更新
4. 四半期ごとにスコアボードで比較 → 成績下位モデルを retire（履歴は保持）、新モデルを投入（origin に出典を記録）

初期モデル3本（テーマ・モメンタム型 / コントラリアン型 / クオリティ・バリュエーション型）をテンプレとして同梱。
いずれも「公開情報のみ」「断定的推奨をしない」の制約文を含みます。編集・retire は自由です。

### 成績の計算仕様（シンプル・厳密性より継続性）
- ピックのリターン = 現在価格（クローズ後はイグジット価格）÷ エントリー価格 − 1。**配当・税・手数料は無視**
- 同一通貨の価格比のため円換算はしない
- ベンチマーク: 米国銘柄 → SPY、日本銘柄（4桁数字 or `.T`）→ 1306（TOPIX 連動 ETF）
- 相対リターン = ピックのリターン − 同期間のベンチマークのリターン
  - ベンチマークの「エントリー時点値」はピック日以前の直近記録。無ければ7日以内の直後の記録で代替、
    それも無ければ**欠損**としてそのピックを相対リターン計算から除外（件数を表示。計算は破綻しない）
- モデルの累積相対リターン = weight 加重平均（計算対象ピックのみ）／ ヒット率 = 相対リターン > 0 のピック比率
- SPY は株価一括更新・ピック取り込み時に自動記録（Finnhub キーがある場合）。1306 は 📈 画面で手動記録

### テスト手順（受け入れ基準の再現）
- **一巡**: 📈 → モデルの実行プロンプトをコピー → AI 回答を貼り戻し → ピック記録 → 資産画面で一括更新 → スコアボードに反映
- **制限リスト警告**: 制限リスト登録銘柄（サンプルは MYCO）を含む回答を貼り戻すと確認ダイアログが出る（除外/包含を選択）
- **retire**: モデル詳細 → retire → スコアボードの「retire済み」フィルタで成績・ピック履歴が引き続き閲覧できる
- **ベンチマーク欠損**: 1306 の記録が無い状態で日本株ピックを追加 → 相対リターンが「欠損」表示になり計算対象外として集計される

## 起動（ローカル確認）

静的サーバーで `sanbo/` を配信するだけです（例: `npx serve sanbo`）。
`scripts/build-feed.mjs` は Node 18+ の組み込み `fetch` を使うため `npm install` 後にそのまま実行できます。

## 使い方の流れ（週次 15 分）

1. **設定** — 必要なら Finnhub キー登録（未設定でも手動運用で全機能が動きます）
2. **資産** — 口座・保有を登録。為替は起動時に自動取得（失敗時は前回値のまま）
3. **テーマ** — 仮説・myEdge・KPI を登録。ウォッチ銘柄はトリガー条件つきで追加
4. **インテル** — 平日は URL/メモを投げ込むだけ。週末に「コンテキストパックをコピー」
   → AI チャットに貼る → 回答全文を貼り戻して取り込み
5. **設定 → JSON エクスポート** — 30 日以上未実施だとダッシュボードに警告が出ます

## データについて

- 全データは端末の localStorage のみに保存。外部送信されるのはティッカーシンボルだけ
- インポートは schemaVersion を確認し、直前の状態を `sanbo.appState.preImportBackup` に自動退避
- 月次スナップショットは起動時に当月分が無ければ自動記録（保有ゼロの間はスキップ）
